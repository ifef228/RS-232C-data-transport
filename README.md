# Курсовая работа. Вариант 38 — программа

Описание только программной части (без отчётов и документации): что сделать, как это должно работать и что использовать.

---

## 1. Суть задания (вариант 38)

- **Топология:** 2 ПК, соединённых нуль-модемным кабелем по RS-232C.
- **Функция:** передача текстовых файлов от источника к приёмнику.
- **Параметры:** скорость и настройки COM-порта задаёт пользователь на каждом ПК.
- **Имя файла:** задаёт источник (отправитель).
- **Защита от ошибок:** циклический код **[15,11]** (на всё содержимое файла при передаче).

---

## 2. Что нужно сделать (только программа)

Реализовать три уровня и связь между ними.

### 2.1. Физический уровень

- **Задача:** работа с последовательным портом (RS-232).
- **Действия:**
  - Открытие/закрытие COM-порта по выбору пользователя.
  - Настройка параметров: скорость (baud rate), биты данных (например 8), стоп-биты, четность (чёт/нечёт/нет).
  - Передача и приём байтов (чтение/запись потока данных).
- **Важно:** на обоих ПК параметры должны совпадать, иначе обмен будет некорректным.

### 2.2. Канальный уровень

- **Задача:** передача данных кадрами с контролем ошибок и повторной передачей при сбоях.
- **Действия:**
  - Формирование кадров по формату из методички (стартовый байт 0xFF, адрес получателя/отправителя, тип кадра, длина, данные, стоповый 0xFF). Для двух ПК адреса можно, например, 0x01 и 0x02.
  - Перед отправкой по порту — кодирование полезных данных **циклическим [15,11]-кодом** (блоки по 11 бит информации → 15 бит кодового слова). При приёме — декодирование и проверка; при ошибке — запрос повтора (например кадр Ret) и повторная передача (ARQ).
  - Управление соединением: установка/разрыв логического канала, подтверждения (ACK), повтор (Ret) при ошибке.
- **Результат:** приложение получает от канального уровня «надёжный» поток байтов (файл уже с учётом кода [15,11] и повторов).

### 2.3. Прикладной уровень

- **Задача:** интерфейс пользователя и логика «передача файла».
- **Действия:**
  - Окно настройки: имя пользователя (ник), выбор COM-порта и параметров (скорость, биты данных, стоп-биты, четность). После настройки — переход к главному окну.
  - Главное окно:
    - У **источника:** поле для ввода имени передаваемого файла (или выбор из каталога), кнопка «Отправить/Передать файл». Имя файла задаёт источник, как в задании.
    - У **приёмника:** кнопка «Принять» (или автоматический приём), выбор каталога для сохранения (по желанию) и сохранение файла под заданным именем.
  - Программа передаёт в канальный уровень содержимое файла (и метаданные: имя файла); канальный уровень дробит данные на кадры, кодирует [15,11], отправляет по порту, обрабатывает ACK/Ret.
- **Результат:** пользователь на одном ПК указывает файл и нажимает «Передать», на другом — принимает и сохраняет тот же текстовый файл.

## 3. Как это должно работать (сценарий)

1. **Запуск:** на каждом ПК запускается своя копия программы (при тесте на одном компьютере — два процесса и виртуальная пара портов).
2. **Настройка:** оба пользователя в окне настроек выбирают свой COM-порт (или путь к виртуальному порту) и выставляют **одинаковые** параметры (скорость, биты данных, стоп-биты, четность). Сохраняют настройки.
3. **Установка связи:** по кнопке типа «Подключиться» программа открывает порт и при необходимости обменивается служебными кадрами установления соединения (по вашему протоколу).
4. **Передача файла:**
   - Источник: пользователь вводит/выбирает путь к текстовому файлу и нажимает «Передать». Имя файла (и при необходимости размер) отправляются в канальном уровне в служебном/первом кадре, далее идёт содержимое, разбитое на блоки, каждый блок защищён [15,11].
   - Приёмник: принимает кадры, декодирует [15,11], при ошибке запрашивает повтор; после успешного приёма сохраняет файл под переданным именем в выбранную папку.
5. **Разрыв:** по кнопке «Отключиться» порт закрывается, соединение разрывается.

Итог: с точки зрения пользователя — два окна, общие настройки порта, у источника ввод имени файла и «Передать», у приёмника — «Принять» и сохранённый текстовый файл.

## 4. Что использовать

| Компонент | Что взять |
|-----------|-----------|
| Язык | **Java** (JDK 11 или 17 удобны; в методичке упоминается Java 5). |
| COM-порт | **jSerialComm** (рекомендуется) или **RXTX** / **javax.comm**. jSerialComm кроссплатформенный, хорошо работает на Windows/macOS/Linux и с виртуальными портами. |
| GUI | **javax.swing** (JFrame, JTextField, JButton, JFileChooser и т.д.). Можно собирать вручную или в NetBeans. |
| Сборка | Maven или Gradle по желанию, либо обычный проект в IDE. |
| Тест без железа | **Виртуальные порты:** на macOS — `socat` (пара pty); на Windows — com0com или аналог. Два процесса программы, каждый открывает свой конец пары — имитация двух ПК. |

### Зависимости (пример для Maven)

```xml
<!-- jSerialComm -->
<dependency>
    <groupId>com.fazecast</groupId>
    <artifactId>jSerialComm</artifactId>
    <version>2.10.4</version>
</dependency>
```

Swing входит в JDK, отдельно подключать не нужно.

## 5. Циклический код [15,11]

- **Назначение:** обнаружение (и при желании исправление) ошибок в блоках при передаче по порту.
- **Параметры:** 15 бит в кодовом слове, 11 бит — информационных, 4 — проверочные. Стандартный циклический код (например, на основе порождающего многочлена для [15,11]).
- **Где применять:** кодировать блоки данных (или целые кадры/их поля) перед отправкой по COM; на приёме — декодировать и проверять. При несовпадении синдрома — считать блок ошибочным и инициировать повтор (ARQ) кадра/блока.
- В методичке и в интернете есть описание циклических кодов и готовые формулы/таблицы для [15,11]; можно реализовать кодирование/декодирование в отдельном классе и вызывать его из канального уровня.

## 6. Рекомендуемая структура проекта (идея)

- **physical** — классы для открытия/закрытия порта, настройки параметров, чтения/записи байтов (обёртка над jSerialComm).
- **datalink** — формат кадров, кодирование/декодирование [15,11], отправка/приём кадров, ACK/Ret, таймауты и повторные передачи (ARQ).
- **application** — окна (настройка, главное), обработчики кнопок, выбор файла, вызов отправки/приёма через канальный уровень.
- **util** — циклический [15,11]: кодер/декодер (и при необходимости конвертация битовых блоков в байты для кадров).

Так вы разделите уровни и сможете тестировать канальный уровень отдельно (например, заглушкой физического уровня) и полный сценарий — двумя процессами на виртуальных портах.

## 7. Краткий чек-лист по программе

- [ ] Физический уровень: открытие/закрытие порта, настройка скорости и параметров, ввод/вывод байтов.
- [ ] Канальный уровень: формат кадров (старт/стоп 0xFF, адреса, тип, длина, данные), обмен кадрами.

- [ ] ARQ: при ошибке [15,11] или потере кадра — запрос повтора (Ret) и повторная передача.
- [ ] Прикладной уровень: окно настроек (порт, параметры), главное окно с вводом имени файла у источника и приёмом/сохранением у приёмника.
- [ ] Согласование имени файла: источник задаёт имя, приёмник сохраняет файл под этим именем.
- [ ] Тест на одном ПК: пара виртуальных портов (socat/com0com), два запуска приложения, передача текстового файла.
