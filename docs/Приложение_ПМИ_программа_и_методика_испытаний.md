# Приложение. Программа и методика испытаний

**Программное изделие:** пересылка текстовых файлов по RS-232 (вариант 38 курсовой работы).  
**Назначение документа:** определить состав испытаний, порядок их проведения и критерии приёмки.

---

## 1. Объект испытаний

Программа на языке Java 17 с графическим интерфейсом (Swing), выполняющая передачу текстовых файлов между двумя узлами по последовательному порту RS-232 с использованием канального протокола (кадры, ACK) и циклического кода **[15,11]** для защиты содержимого файла.

**Точка входа:** `ru.mgtu.bauman.rs232.Main`.

---

## 2. Условия проведения испытаний

### 2.1. Аппаратные и программные средства

| Средство | Требование |
|----------|------------|
| ПЭВМ | Совместимая с Java 17 и ОС macOS / Windows / Linux (по ТЗ — macOS) |
| JDK | 17 |
| Сборка | Apache Maven 3.x |
| Зависимость | jSerialComm (подтягивается Maven в `target/lib/`) |

### 2.2. Вариант «один компьютер» (рекомендуется для лабораторной проверки)

- Утилита **socat** (macOS/Linux) или пара виртуальных портов **com0com** (Windows).
- Два одновременных процесса приложения.
- Пара виртуальных последовательных портов, соединённых «кроссом» (см. `README-VIRTUAL-PORTS.md` в репозитории).

### 2.3. Вариант «два ПК»

- Два компьютера с COM-портами (или USB–UART).
- Нуль-модемный кабель или эквивалентное соединение линий TD/RD и служебных сигналов по методичке.
- Одинаковые параметры UART на обоих концах (скорость, формат кадра).

---

## 3. Программа испытаний

### Испытание 1. Сборка и запуск

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| 1.1 | В каталоге проекта выполнить `mvn -q package` | Сборка завершена без ошибок; в `target/` появляются классы и `lib/*.jar` |
| 1.2 | Выполнить `java -cp "target/classes:target/lib/*" ru.mgtu.bauman.rs232.Main` | Открывается главное окно приложения |

**Критерий приёмки:** испытание 1 считается пройденным при успешном выполнении 1.1 и 1.2.

---

### Испытание 2. Подключение виртуальной пары портов (один ПК)

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| 2.1 | Запустить `socat` (см. `README-VIRTUAL-PORTS.md`) | В терминале отображаются созданные устройства; процесс `socat` активен |
| 2.2 | В первом окне приложения: режим **Приёмник**, порт `/tmp/vcomB` (или второй конец пары), адреса **2** и **1**, каталог для сохранения, **Подключить порт** | В логе сообщение о запуске приёма; ошибок открытия порта нет |
| 2.3 | Во втором окне: режим **Передатчик**, порт `/tmp/vcomA`, адреса **1** и **2**, **Подключить порт** | Порт открыт без ошибки |

**Критерий приёмки:** оба порта открыты, параметры UART на обоих окнах совпадают.

---

### Испытание 3. Передача текстового файла

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| 3.1 | Подготовить небольшой текстовый файл (например, несколько строк UTF-8) | Файл доступен на диске |
| 3.2 | На передатчике указать путь к файлу, нажать **Передать файл** | В логе этапы Link, META, передача данных, END, Uplink; сообщение «Готово» |
| 3.3 | Проверить каталог приёмника | Файл с **тем же именем**, что у исходного, содержимое совпадает с исходным (побайтово для текстового файла) |

**Критерий приёмки:** файл получен без искажений; в логе приёмника — сообщения о приёме кадров и сохранении файла.

---

### Испытание 4. Повторная передача без переподключения порта

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| 4.1 | Не закрывая порты, на передатчике выбрать другой файл и снова **Передать файл** | Вторая передача завершается успешно |
| 4.2 | Проверить каталог приёмника | Второй файл сохранён корректно |

**Критерий приёмки:** обе передачи успешны без **Отключить** / **Подключить** между ними.

---

### Испытание 5. Согласование адресов и параметров

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| 5.1 | Установить на обоих узлах **одинаковые** скорость и формат кадра, корректные пары адресов (1↔2 и 2↔1) | Передача выполняется (испытание 3) |
| 5.2 | Намеренно перепутать локальный/удалённый адрес на одном из узлов | Передача не завершается (нет ACK); в логе передатчика — ошибка ожидания ACK |

**Критерий приёмки:** при корректных адресах обмен есть; при ошибке конфигурации — отказ с диагностическим сообщением.

---

### Испытание 6. Логирование (опционально)

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| 6.1 | Запуск из терминала | В консоли дублируются строки с префиксами `[GUI]`, `[PHY TX/RX]`, `[канал]` при обмене |

**Критерий приёмки:** при необходимости отладки обмен виден в терминале.

## 4. Методика испытаний

1. Испытания проводят в порядке, указанном в разделе 3 (при необходимости испытание 5 можно совместить с проверкой после испытания 3).
2. Результаты фиксируют в протоколе: дата, версия сборки, конфигурация (один ПК / два ПК), для каждого пункта — «пройдено» / «не пройдено», замечания.
3. Программное изделие считают **готовым к защите**, если пройдены испытания **1–4** в типовой конфигурации (виртуальные порты или два ПК), а требования ТЗ по передаче текстового файла и использованию кода **[15,11]** подтверждены анализом кода и успешным испытанием 3.
