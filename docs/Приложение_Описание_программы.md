# Приложение. Описание программы

**Полное наименование:** программа пересылки текстовых файлов по интерфейсу RS-232 (локальная безадаптерная сеть, 2 узла).  
**Вариант задания:** 38 (методические указания к курсовой работе по дисциплине «Сетевые технологии в АСОИУ»).  
**Язык реализации:** Java 17.  
**Точка входа:** класс `ru.mgtu.bauman.rs232.Main`.

---

## 1. Назначение и общие сведения

Программа предназначена для передачи **текстовых файлов** от узла-источника к узлу-приёмнику по последовательному каналу (RS-232). На физическом уровне используется последовательный порт; на канальном — собственный протокол кадрирования с подтверждениями; содержимое файла перед передачей по кадрам данных защищается **циклическим кодом [15,11]**.

Пользователь задаёт номер COM-порта и параметры обмена через меню; имя передаваемого файла — на передающем ПК, каталог для сохранения — на принимающем (в соответствии с ТЗ).

## 2. Структура программы и состав модулей

Программа организована по уровням взаимодействия с пользователем и среде передачи:

| Уровень | Пакет / классы | Назначение |
|---------|----------------|------------|
| Прикладной | `ru.mgtu.bauman.rs232.app` | `MainFrame` — интерфейс пользователя; `FileTransferService` — сценарий передачи/приёма файла |
| Канальный | `ru.mgtu.bauman.rs232.datalink` | `DataLinkLayer`, `WireCodec`, `Frame`, `ProtocolConstants` — кадры, очереди, ACK, повторная передача |
| Физический | `ru.mgtu.bauman.rs232.physical` | `SerialPhysicalLayer` — работа с COM-портом через библиотеку jSerialComm |
| Защита данных | `ru.mgtu.bauman.rs232.codec` | `CyclicCode1511`, `PayloadCodec1511` — циклический [15,11], битовая упаковка |
| Вспомогательный | `ru.mgtu.bauman.rs232.util` | `AppLog` — дублирование сообщений в консоль |

## 3. Внешние интерфейсы

### 3.1. Пользовательский интерфейс

- Графическое окно (Swing): выбор режима «Передатчик» / «Приёмник», поля COM-порта, скорости, бит данных, стоп-битов, чётности, локального и удалённого адреса, путь к файлу или каталогу.
- Кнопки: «Подключить порт», «Отключить», действие передачи / справка для приёмника.
- Область текстового лога событий.

### 3.2. Аппаратный / программный интерфейс

- Последовательный порт: доступ через **jSerialComm** (реализация физического уровня).
- Параметры UART задаются пользователем и должны совпадать на обоих узлах.

## 4. Логическая схема данных

### 4.1. Кадр канального уровня (после физического кодирования)

Бинарная последовательность: `0xFF` | `DST` | `SRC` | `TYPE` | `SEQ` | `LEN_HI` | `LEN_LO` | *экранированное тело* | `0xFF`.

Экранирование в поле «тело»: байт `0xFF` → `0xFE 0x01`, байт `0xFE` → `0xFE 0x02`. Поле `LEN` — длина **неэкранированной** полезной нагрузки в байтах.

### 4.2. Типы кадров (`ProtocolConstants`)

| Код | Константа | Назначение |
|-----|-----------|------------|
| 0x01 | FT_LINK | Установление логического соединения |
| 0x02 | FT_UPLINK | Разрыв логического соединения (конец сессии) |
| 0x03 | FT_I | Информационный кадр (фрагмент закодированных данных файла) |
| 0x04 | FT_ACK | Подтверждение приёма (в полезной нагрузке — номер `seq`) |
| 0x05 | FT_RET | Запрос повтора (зарезервировано в логике приёма) |
| 0x06 | FT_META | Метаданные файла: имя (UTF-8, нулевой байт) + 4 байта размера исходного файла (big-endian) |
| 0x07 | FT_END | Конец передачи содержимого файла |
| 0x08 | FT_NOTIFY | Текстовое уведомление удалённому абоненту |

### 4.3. Циклический код [15,11]

- Информационные блоки по **11 бит**; кодовое слово **15 бит**.
- Порождающий многочлен степени 4: **g(x) = x⁴ + x + 1** (0x13).
- Систематическое кодирование: остаток от деления **m(x)·x⁴** на **g(x)** в поле GF(2) дописывается как проверочные биты.
- При декодировании вычисляется синдром; при ненулевом синдроме — ошибка (в сценарии передачи файла возможна передача уведомления на прикладном уровне).

Поток байтов файла преобразуется в битовый поток, режется на блоки по 11 бит (последний блок дополняется до 11 бит), кодируется, затем кодируется потоком байтов для передачи в кадрах `FT_I`.

## 5. Алгоритм работы (прикладной уровень)

### 5.1. Передача (узел-источник)

1. Открыть порт, запустить канальный уровень.
2. Отправить кадр `Link` с надёжной доставкой (ожидание ACK).
3. Отправить кадр `META` с именем и размером исходного файла в байтах.
4. Прочитать файл в память, выполнить кодирование [15,11], разбить результат на фрагменты не длиннее `MAX_CHUNK`, отправить каждый фрагмент в кадрах `FT_I` с ACK.
5. Отправить `END`, затем `Uplink` с ACK.

### 5.2. Приём (узел-приёмник)

1. Открыть порт, запустить канальный уровень и цикл приёма в фоновом потоке.
2. На каждый принятый кадр, адресованный данному узлу, отправлять ACK по правилам канального уровня.
3. Обработать `META` — подготовить каталог и имя файла; накопить полезную нагрузку кадров `FT_I`.
4. По `END` выполнить декодирование [15,11], записать файл в указанный каталог.
5. По `Uplink` завершить сессию и **снова перейти в ожидание** следующей передачи (без обязательного закрытия порта).

---

## 6. Надёжная доставка на канальном уровне

Для кадров с флагом «надёжная отправка» после передачи кадра ожидается кадр `ACK` с тем же номером `seq`, что и у отправленного кадра. При отсутствии подтверждения в течение таймаута выполняется повторная отправка (ограниченное число попыток).

---

## 7. Ограничения

- Весь файл загружается в оперативную память целиком; очень большие файлы могут потребовать доработки (потоковая передача).
- Скорость ограничена параметрами последовательного порта и накладными расходами протокола и кодирования.
