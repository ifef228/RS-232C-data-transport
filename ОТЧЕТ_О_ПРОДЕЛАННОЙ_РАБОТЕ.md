# Отчёт о проделанной работе

**Тема:** программа пересылки текстовых файлов по локальной безадаптерной сети (RS-232, вариант 38).  
**Дисциплина:** «Сетевые технологии в АСОИУ».  
**Основание:** методические указания к курсовой работе, техническое задание на программный продукт (группа РТ5-61Б, вариант №38).

---

## 1. Цель работы

Разработать программное обеспечение, реализующее протоколы взаимодействия объектов **физического**, **канального** и **прикладного** уровней для двух узлов, соединённых по интерфейсу **RS-232C** нуль-модемно; обеспечить **передачу текстовых файлов** с защитой полезной нагрузки **циклическим кодом [15,11]**; параметры COM-порта и номер порта задаются пользователем через интерфейс; имя файла задаётся на передающей стороне, каталог сохранения — на принимающей.

---

## 2. Выполненные задачи

| Требование ТЗ / методички | Реализация |
|---------------------------|------------|
| Физический уровень: параметры COM-порта, установление и разъединение канала | Класс `SerialPhysicalLayer`: открытие порта (jSerialComm), настройка скорости, бит данных, стоп-битов, чётности, отключение аппаратного flow control, чтение/запись байтов |
| Канальный уровень: логическое соединение, кадры, порядок блоков, контроль ошибок, разрыв соединения | Кадры по формату с разделителем `0xFF`, поля адресов, типа, `seq`, длины, полезной нагрузки с экранированием; типы `Link`, `META`, `I`, `END`, `Uplink`, `ACK`, `Ret`, `NOTIFY`; надёжная доставка с ожиданием **ACK** и повтором при таймауте |
| Циклический код [15,11] | Классы `CyclicCode1511`, `PayloadCodec1511`: кодирование потока байтов файла блоками по 11 бит в 15-битные кодовые слова, декодирование с проверкой синдрома |
| Прикладной уровень: меню, режимы, выбор порта и параметров, файл / каталог | Swing: `MainFrame` — режимы «Передатчик» / «Приёмник», выбор файла и каталога, подключение порта, передача и фоновый приём |
| Уведомление пользователей об ошибках | Диалоги `JOptionPane`, сообщения в лог; опционально `FT_NOTIFY` для текста удалённому абоненту |
| Среда: Java 17, macOS (по ТЗ) | Проект Maven, `maven.compiler` 17, зависимость jSerialComm |

Дополнительно по ходу разработки:

- Дублирование логов в **терминал** (`AppLog`, префиксы `[GUI]`, `[PHY TX/RX]`, `[канал]`) для отладки без привязки только к окну приложения.
- Рекомендуемые по умолчанию **виртуальные порты** `/tmp/vcomA` / `/tmp/vcomB` и адреса **1↔2** при переключении режима (соответствие инструкции с `socat`).
- Каталог приёма по умолчанию — **текущая рабочая директория** запуска (`user.dir`), обычно папка проекта курсача.
- Исправление режима **чтения порта** (semi-blocking вместо полного блокирующего ожидания «порции» байт), из-за чего короткие кадры не доходили до разборщика.
- Приёмник после завершения сессии (**Uplink**) **не завершает** цикл ожидания — можно передавать **несколько файлов** подряд без переподключения порта.
- Документация: `README.md`, `README-VIRTUAL-PORTS.md` (виртуальные порты, порядок запуска приёмник → передатчик).

## 3. Архитектура программы

```
┌─────────────────────────────────────────────────────────┐
│  Прикладной уровень: MainFrame, FileTransferService      │
│  (файл, каталог, сценарий Link→META→I→END→Uplink)        │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  Канальный уровень: DataLinkLayer, WireCodec, Frame      │
│  (кадрирование, ACK, очереди app/ack)                   │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  Физический уровень: SerialPhysicalLayer               │
│  (jSerialComm: COM-порт, readBytes / writeBytes)         │
└─────────────────────────────────────────────────────────┘

         Кодек: CyclicCode1511 + PayloadCodec1511
         (встраивается в сценарий передачи файла)
```

**Структура каталогов исходного кода:**

| Путь | Назначение |
|------|------------|
| `ru.mgtu.bauman.rs232.Main` | Точка входа |
| `...app.MainFrame` | GUI |
| `...app.FileTransferService` | Логика передачи/приёма файла |
| `...physical.SerialPhysicalLayer` | RS-232 |
| `...datalink.*` | Протокол кадров и надёжная доставка |
| `...codec.*` | Код [15,11] |
| `...util.AppLog` | Вывод в консоль |

## 4. Протокол обмена (кратко)

1. **Link** — установление логического соединения (подтверждается ACK).
2. **META** — имя файла (UTF-8, нулевой терминатор) и размер исходного файла в байтах.
3. **I** — фрагменты **закодированного** [15,11] содержимого (размер фрагмента ограничен константой `MAX_CHUNK`).
4. **END** — конец потока данных файла; на приёмнике выполняется декодирование и запись файла.
5. **Uplink** — разрыв логической сессии (ACK); приёмник возвращается в ожидание следующей передачи.

Порождающий многочлен циклического кода: **g(x) = x⁴ + x + 1** (0x13), систематическое кодирование 11 информационных бит в 15-битное слово.

## 5. Средства разработки и сборки

- **Язык:** Java 17.  
- **Сборка:** Apache Maven.  
- **Библиотека последовательного порта:** jSerialComm 2.10.4.  
- **Интерфейс:** javax.swing.

Команда сборки и запуска:

```bash
mvn -q package
java -cp "target/classes:target/lib/*" ru.mgtu.bauman.rs232.Main
```

## 6. Порядок испытаний (рекомендуемый)

1. На одном ПК: утилита **socat** создаёт пару виртуальных портов (`/tmp/vcomA` и `/tmp/vcomB` — см. `README-VIRTUAL-PORTS.md`).
2. Запускаются **два процесса** приложения.
3. **Приёмник:** каталог сохранения, порт `vcomB`, адреса локальный 2 / удалённый 1 → «Подключить порт» (ожидание стартует автоматически).
4. **Передатчик:** файл, порт `vcomA`, адреса 1 / 2 → «Подключить порт» → «Передать файл».
5. Проверка: файл появляется в каталоге приёмника с тем же именем; при необходимости — повторная передача другого файла без отключения порта.

На двух ПК — тот же порядок адресов и согласованные параметры UART; вместо виртуальных портов — физические COM и нуль-модемный кабель.

## 7. Ограничения и возможное развитие

- Файл целиком читается в память; для очень больших файлов целесообразна потоковая передача.
- Скорость ограничена COM и объёмом служебных данных протокола и [15,11].
- Расширения: явный фильтр «только текстовые расширения» в диалоге выбора файла, сохранение настроек порта в файл конфигурации.

---

## 8. Заключение

В рамках курсового проекта по варианту 38 реализовано приложение на Java 17 с разделением на физический, канальный и прикладной уровни, передачей текстовых файлов между двумя узлами по RS-232 и защитой данных циклическим кодом **[15,11]**. Требования технического задания и методических указаний по структуре уровней и сценарию передачи учтены. Предусмотрены удобство тестирования на одном компьютере (виртуальные порты), логирование и повторные передачи без обязательного переподключения порта после первого файла.
